# 学业预警系统答辩讲稿（结合当前代码实现）

> 适用场景：课堂答辩/毕设答辩现场演示。  
> 建议总时长：10~12 分钟（讲解 8 分钟 + 问答 2~4 分钟）。

---

## 1. 开场（30~45 秒）

**你可以这样说：**

“各位老师好，我的课题是《学业预警系统的设计与实现》。系统目标是基于学生日常学习行为和测评结果，自动计算学业风险等级，辅助教师尽早发现问题并进行干预。  
系统采用前后端分离架构：前端使用 Vue3 + Element Plus，后端使用 Spring Boot + Spring Security + MyBatis，数据库使用 MySQL。核心能力是‘任务驱动的数据采集 + 逻辑回归风险评估 + 多角色可视化看板’。”

---

## 2. 系统架构页讲解（1 分钟）

**你可以这样说：**

“系统分三类角色：学生、教师、管理员。登录后通过路由守卫做角色级访问控制，避免跨角色越权访问。  
后端统一提供 REST API，前端以 Axios 封装请求；数据库层按模块拆分 Mapper，记录任务、提交、学习行为、风险记录与用户风险状态。  
其中风险计算由教师端触发每日计算，也支持学生提交后实时刷新个人风险，形成‘批处理 + 实时更新’的混合预警机制。”

**技术点强调：**
- Vue Router 按角色分离页面并在 `beforeEach` 做鉴权。
- Spring Security + JWT 做认证鉴权。
- MyBatis 直接写 SQL，风险特征由多表聚合查询生成。

## 3. 现场演示脚本（按“点击哪里 + 说什么”组织）

---

### 3.1 登录与角色隔离（约 1 分钟）

#### 操作步骤
1. 打开登录页 `/login`。
2. 输入账号、密码，选择角色下拉框（学生端/教师端/管理员端）。
3. 点击“登录”。
4. 演示：切换一个不匹配角色的路由地址，观察被重定向回登录页。

#### 你可以这样说
“登录时除用户名和密码外，还显式选择角色。后端返回 JWT 和角色信息，前端写入本地缓存后跳转到角色首页。  
路由守卫会检查 token 与角色，如果访问了非本角色路径，会被立即拦截。这一设计降低了前端误操作导致的越权风险。”

#### 可补充的技术话术
“这里体现了认证与授权分离：认证解决‘你是谁’，授权解决‘你能访问什么’。”

---

### 3.2 学生端：任务完成 → 风险变化（约 2.5 分钟）

#### 操作步骤
1. 左侧菜单点击：`任务列表`。
2. 选择一个“作业/考试”任务，点击进入。
3. 随机提交答案，观察提交成功提示（会返回得分和最新风险等级）。
4. 返回任务列表，再点击 `学业预警` 页面查看等级。
5. 可额外展示顶部/右下角提醒弹窗：待完成任务提醒与连续 7 天 HIGH 风险提醒。

#### 你可以这样说
“学生每次完成作业、视频、考试后，系统会更新当日学习行为聚合数据，比如作业完成数、视频学习时长、平均分，再调用风险服务实时重算当前风险分。  
所以学生端不是静态展示，而是‘行为发生后立刻反馈风险等级’，这有助于学生形成即时改进闭环。  
另外系统支持高风险连续告警：如果最近 7 天每天都是 HIGH，就会主动弹窗提醒尽快联系教师。”

#### 可补充技术话术
“我们将行为数据拆成原子记录表和每日聚合表，既保留明细，也保证风险计算性能。”

### 3.3 教师端：触发风险计算与看板（约 2.5 分钟）

#### 操作步骤
1. 切换教师账号登录。
2. 点击左侧 `触发风险计算`。
3. 在日期选择器选择日期，点击“立即计算”。
4. 展示结果区：
   - 更新人数 `updatedCount`；
   - 模型参数（bias、各特征权重、阈值、precision/recall/f1/accuracy）。
5. 点击 `预警看板`，展示风险分布、趋势、近 7 天预警清单。

#### 你可以这样说
“教师侧以‘天’为单位进行批量风险计算，接口会先训练当天可用的逻辑回归模型，再对所有学生进行评分并写入风险记录。  
看板展示的是已落库的结果，不是临时计算，因此可追溯、可统计。  
模型参数公开展示，便于教师理解‘当前预警更受哪些行为因子影响’，这也是可解释性设计的一部分。”

#### 可补充技术话术
“风险趋势图来自 `risk_record` 的历史数据，支持后续做教学干预效果对比。”

### 3.4 管理员端：全局治理（约 1.5 分钟）

#### 操作步骤
1. 切换管理员账号。
2. 点击 `用户管理` 查看用户角色与风险等级。
3. 点击 `全局看板` 查看全局风险分布与趋势。

#### 你可以这样说
“管理员端关注的是系统级运营视角：用户管理保证角色与账号可控；全局看板关注群体风险结构变化，为教学管理提供决策依据。  
三端职责分离后，系统既能满足学生自助改进，也能支持教师干预和管理层治理。”

## 4. 重点讲清楚：逻辑回归算法是如何实现的（答辩核心）

下面建议你按“数据 → 特征 → 训练 → 阈值 → 线上推断”五步讲，老师一般很容易听懂。

### 4.1 数据输入（训练集来源）

**讲稿：**
“训练数据不是手工标注，而是从近 60 天历史行为中自动构造。系统读取学生每日聚合行为（登录次数、视频时长、作业提交、平均成绩）并结合考试通过情况，形成样本。”

### 4.2 特征工程（5 维核心特征）

**讲稿：**
“模型输入是 5 维特征：
1) `scoreGap = 1 - avgScore/100`（成绩缺口）  
2) `examFailRate`（考试失败率）  
3) `homeworkMissRate`（作业缺交率）  
4) `normalizedVideoGap`（视频学习不足度）  
5) `normalizedLoginGap`（登录活跃不足度）

这些特征都被统一成‘值越大风险越高’的方向，便于线性模型学习。”

### 4.3 标签构造（弱监督规则）

**讲稿：**
“标签 y 由次日状态推导：如果次日出现至少 2 个风险信号（低分、作业缺交、视频不足、考试失败率高），则标注为 1，否则为 0。  
这个做法本质是规则生成伪标签，再用逻辑回归学习连续概率，兼顾可落地与可解释性。”

### 4.4 模型训练（手写梯度下降）

**讲稿：**
“训练时先做标准化，再通过批量梯度下降优化交叉熵目标，并加入 L2 正则抑制过拟合。  
具体流程是：
- 计算 `pred = sigmoid(w·x + b)`；
- 误差 `err = pred - y`；
- 累积梯度更新 `w,b`；
- 训练轮数固定为 500 轮，学习率 0.08，L2 系数 1e-3。  
最终得到权重、偏置和标准化参数。”

### 4.5 阈值与风险分层

**讲稿：**
“模型输出是 0~1 的风险概率。系统不是固定阈值，而是在训练样本上网格搜索阈值使 F1 最优：
- 先找中风险阈值 `mediumThreshold`；
- 再找高风险阈值 `highThreshold`，并保证高阈值大于中阈值。  
线上推断时：`score >= high` 判 HIGH，`score >= medium` 判 MEDIUM，否则 LOW。”

### 4.6 线上推断与落库

**讲稿：**
“每天计算时，系统对每位学生做一次推断，得到 `riskScore` 与 `riskLevel`，并把特征和阈值细节写入 `detail_json`。  
这样既能追溯单次判断依据，也便于后续审计和模型迭代。”

## 6. 答辩高频问题清单（含建议回答）

### Q1：为什么选逻辑回归而不是随机森林/神经网络？
**建议回答：**
“当前场景样本规模中等，且教学系统更强调可解释性。逻辑回归能直接给出每个特征的方向和权重，便于教师理解与信任。后续数据规模扩大后可引入更复杂模型做对比实验。”

### Q2：你的标签不是人工标注，会不会不准？
**建议回答：**
“是的，它是弱监督标签。我们用次日风险信号规则生成伪标签，优点是低成本可持续更新。下一步可以结合教师反馈形成人工校准集，做半监督或主动学习提高标签质量。”

### Q3：如何避免特征量纲不一致影响训练？
**建议回答：**
“训练和推断都使用同一套均值/标准差做标准化，标准差过小时置 1 防止数值不稳定。”

### Q4：为什么用 F1 选阈值？
**建议回答：**
“预警任务通常类别不平衡，单看准确率会掩盖问题。F1 能同时考虑查准率和查全率，更贴近‘尽量发现风险且控制误报’的目标。”

### Q5：模型会过拟合吗？
**建议回答：**
“我们用了 L2 正则，并限制特征维度在 5 维，复杂度较低；另外每天重训可降低概念漂移风险。后续可以增加时间切片验证和回测。”

### Q6：系统的实时性如何？
**建议回答：**
“有两层机制：教师端按天批量计算保证全量一致性；学生端提交后实时刷新个人风险保证交互及时性。”

### Q7：如果历史数据太少怎么办？
**建议回答：**
“当样本不足时会回退到默认参数配置，确保系统可用性，不会因为训练失败导致服务不可用。”

### Q8：如何保证安全性？
**建议回答：**
“通过 JWT + Spring Security 做认证鉴权，前端路由守卫做二次拦截；后端接口按角色注解限制访问。”

## 7. 现场演示小贴士（避免翻车）

1. **提前准备 3 个账号**：学生、教师、管理员。  
2. **先演示教师触发计算**，再切学生看风险变化，逻辑最顺。  
3. 若网络慢，优先演示“功能闭环”而非所有页面细节。  
4. 回答算法问题时，始终围绕：
   - 特征从哪来；
   - 标签怎么构造；
   - 梯度下降怎么训练；
   - 阈值怎么定；
   - 结果如何落库可追溯。
