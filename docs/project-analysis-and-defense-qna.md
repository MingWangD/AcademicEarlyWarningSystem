# 学业预警系统项目深度分析与答辩问答（评委视角）

## 1. 项目定位与总体评价

该项目是一个面向**学生、教师、管理员三角色**的学业预警平台，技术路线为 **Vue + Spring Boot + MyBatis + MySQL + JWT**。系统从“任务发布—学习行为采集—风险计算—看板展示”形成了完整业务闭环，具备毕业设计项目中较强的工程完整性与可演示性。

从代码实现看，系统具备以下工程亮点：

1. **角色权限闭环完整**：登录后按角色访问不同接口与页面，后端通过 `@PreAuthorize` 控制教师接口访问权限。
2. **数据链路完整**：作业、考试、视频行为均可落库，并通过 `student_daily_activity` 聚合每日行为特征。
3. **预警结果可追溯**：每次计算均记录到 `risk_record`，支持风险趋势、近7天预警、连续高风险判断。
4. **测试意识较好**：包含认证流、教师风险接口等集成测试文件与性能测试脚本。

## 2. 系统架构与核心业务流

### 2.1 后端分层

- `api`：REST 接口层（如教师侧 `/api/v1/teacher/risk/calc`）。
- `service`：业务层（任务发布、评分、风险计算、看板汇总）。
- `mapper`：数据访问层（MyBatis 注解 SQL，含行为聚合、风险趋势等查询）。
- `entity/dto/security`：实体、参数对象、JWT 安全组件。

### 2.2 前端模块

- 学生端：任务、作答、预警信息。
- 教师端：发布任务、活动汇总、风险计算、仪表盘。
- 管理员端：用户管理、全局风险看板。

### 2.3 主业务流

1. 教师发布作业/考试任务。
2. 学生完成作业、考试、视频学习。
3. 系统更新 `student_daily_activity`（登录、视频分钟、作业提交、成绩等）。
4. 教师触发风险计算，系统更新 `risk_record` 与用户 `risk_level`。
5. 仪表盘展示风险分布、趋势和重点学生。

## 3. 逻辑回归算法在项目中的“设计”和“当前运行状态”

> 这一部分是答辩重点，建议你在答辩中主动说明“算法模块已实现，但生产路径当前仍是规则法”。

### 3.1 逻辑回归模块如何工作（代码已实现）

`RiskAnalysisService` 已实现较完整的逻辑回归训练+推理链路：

1. **特征构建**：将原始行为转为 5 维风险特征（成绩差距、考试挂科率、作业缺交率、视频缺口、登录缺口）。
2. **样本构造**：通过历史日行为构造训练样本，并使用“次日风险信号规则”生成监督标签。
3. **标准化**：计算 `means/stds`，对训练和预测特征做标准化。
4. **训练**：梯度下降 + L2 正则优化权重和偏置。
5. **阈值搜索**：在候选区间搜索最优阈值（以 F1 为目标），得到 `mediumThreshold` 和 `highThreshold`。
6. **评估与输出**：输出 precision/recall/f1/accuracy；推理阶段通过 `sigmoid(z)` 产出风险概率，再映射到 LOW/MEDIUM/HIGH。

### 3.2 当前线上业务如何运行（实际生效路径）

当前教师触发 `/teacher/risk/calc` 时，`TeacherService.calculateRisk` 使用的是**学分规则法**：

- 学分来自作业均分 40% + 考试均分 60%，折算满分 4 分。
- `credit >= 3` 为 LOW，`2~3` 为 MEDIUM，`<2` 为 HIGH。
- 风险分按 `1 - credit/4` 计算并写入 `risk_record`。

学生侧 `StudentService.refreshRiskLevel` 也采用同一学分规则。

### 3.3 评委可能会问到的“算法一致性”问题

项目存在“**算法已实现但未接入主流程**”的现实状态：

- 逻辑回归服务在代码中存在，且训练评估逻辑完整；
- 但风险计算主接口目前返回的是“按学分规则完成风险更新”。

**建议答辩表述**：
“当前版本为保证可解释性与稳定性，生产路径默认规则法；逻辑回归模块已完成并可平滑切换，后续将通过配置开关按课程/学院逐步灰度上线。”

## 4. 评委老师提问（含参考回答）

### 问题1：为什么你要做三角色系统，而不是只做一个学生端？
**参考回答**：
学业预警是“教学管理”问题，不是单点应用。学生端只能看结果；教师端负责任务组织和干预；管理员端负责全局治理和资源配置。三角色设计保证了“识别—反馈—管理”闭环，符合真实教学场景。

### 问题2：你是如何定义“风险”的？阈值依据是什么？
**参考回答**：
当前主流程采用学分规则：作业与考试按 4:6 加权，折算 4 分学分后做三档划分（<2、2~3、3~4）。其优点是可解释、易落地。与此同时，我实现了逻辑回归版本，用于概率化输出和阈值搜索，后续会根据历史数据评估结果切换或融合。

### 问题3：逻辑回归标签是人工标注吗？如何避免标签噪声？
**参考回答**：
当前标签由“次日风险信号规则”自动生成（如低分、缺交、低活跃、考试高挂科率）。这属于弱监督方案，能快速启动。后续计划加入教师干预结果、补考结果和学期终绩等真实结果标签，降低噪声并提升泛化能力。

### 问题4：如果老师问“为什么某学生是高风险”，你的系统能解释吗？
**参考回答**：
目前规则法可直接解释为“学分不足”。逻辑回归路径中，我已在 `detail_json` 输出特征与阈值信息，后续会把特征贡献可视化（如雷达图、条形图），把“结果可见”升级为“原因可见”。

### 问题5：这个系统在工程上最关键的可扩展点是什么？
**参考回答**：
我把风险数据单独存成 `risk_record` 历史表，支持趋势分析、回溯和模型复训；另外前后端分离后，未来可替换风险引擎（规则法→逻辑回归→集成模型）而不影响页面结构。

### 问题6：你如何验证系统不是“能跑但不可信”？
**参考回答**：
我做了接口集成测试（认证流程、教师风险接口）和部分前端 E2E/性能测试脚本。下一步会补上离线评估报表（混淆矩阵、AUC、分层召回）与线上监控（每日分布漂移），建立“开发—评估—上线”闭环。

### 问题7：你觉得当前版本最大的不足是什么？
**参考回答**：
一是主流程尚未正式接入逻辑回归；二是解释性展示还偏弱；三是干预策略推荐仍以人工经验为主。后续我会优先做“算法开关 + 解释性可视化 + 干预模板引擎”。

## 5. 可直接用于答辩开场的“简短项目介绍”（30~45秒）

我的毕业设计是一个学业预警系统，面向学生、教师和管理员三类用户。系统通过采集学生的作业、考试、视频学习和活跃度数据，生成风险等级并在教师与管理员看板中展示，用于提前发现可能掉队的学生。项目采用 Vue + Spring Boot + MySQL，已经形成从任务发布到预警反馈的完整闭环。

在算法上，我实现了逻辑回归流程：先做特征构建和标准化，再用梯度下降训练权重，通过 sigmoid 输出风险概率，并用阈值将概率映射为高/中/低风险。目前线上默认使用可解释的学分规则法，逻辑回归模块已具备接入条件。

## 6. 答辩策略建议（实战）

1. **先讲工程价值，再讲算法细节**：先让评委认可“系统是可运行的完整产品”。
2. **主动承认边界**：明确说明“算法已实现，当前默认规则法”，这会让你更诚实可信。
3. **准备一页“演进路线图”**：规则法（已上线）→ 逻辑回归（灰度）→ 干预推荐（目标）。
4. **强调可解释性**：教育场景里，可解释比一味追求复杂模型更容易获得认可。
